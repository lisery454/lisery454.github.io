图解设计模式

设计模式就是如何组织代码的艺术，类似一个房间里面摆放东西，如何才能把东西放得方便使用和效率。

(本文只是为了自己复习，如果要深入了解可以参考：https://refactoringguru.cn/design-patterns )

在gof的书中，分为三大类，即创建型模式，结构型模式，行为模式。

创建型模式

工厂方法 Factory Method

对某个接口的类的创建的封装，方便在创建的时候进行某些操作。

工厂方法将创建产品的代码与实际使用产品的代码分离， 从而能在不影响其他代码的情况下扩展产品创建部分代码。

对象池也算一种特殊的工厂模式。

抽象工厂 Abstract Factory

抽象工厂为你提供了一个接口， 可用于创建每个系列产品的对象。 只要代码通过该接口创建对象， 那么你就不会生成与应用程序已生成的产品类型不一致的产品。

类似于工厂方法，只是把很多需要一起用的产品放到一个工厂来使用

生成器 Builder

封装构造函数，方便用户构造。因为有些类构造很复杂，要先怎样后怎样。

原型模式 Prototype

很多语言都内置的Clone接口，方便复制

单例模式 Singleton

方便全局访问，一般游戏中用到很多。

结构型模式

适配器模式 Adapter

封装接口，方便自己使用。同时也隔离了依赖

桥接模式 Bridge

将调用行为分为两层函数，最普遍的就是springboot中Controller调Service类似的。

组合模式 Composite

组合模式为你提供了两种共享公共接口的基本元素类型： 简单叶节点和复杂容器。 容器中可以包含叶节点和其他容器。 这使得你可以构建树状嵌套递归对象结构。

装饰模式 Decorator

用类来修饰另一个类

外观模式 Facade

为复杂的API提供一个公开接口

享元模式 Flyweight

同一个数据在内存中存储过多就可以使用这个模式

代理模式 Proxy

代理类对被代理类的一些接口进行自己的优化和处理。

行为模式

责任链模式 Chain of Responsibility

对一个行为的分步处理

命令模式 Command

把具体的行为封装成类

迭代器模式 Iterator

将集合的遍历行为抽取为单独的迭代器对象。

中介者模式 Mediator

把多个类之间的交互行为变成多个类对一个类的交互行为

备忘录模式 Memento

允许在不暴露对象实现细节的情况下保存和恢复对象之前的状态

观察者模式 Observer

观察状态变化，自动触发行为

状态模式 State

有限状态机

策略模式 Strategy

类似命令模式，但是目的不同

模板方法模式 Template Method

抽象类中声明接口和默认实现，子类中可以修改默认实现

访问者模式 Visitor

将算法与其所作用的对象隔离开来
